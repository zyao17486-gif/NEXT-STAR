# Next Star — AI 产品架构全景文档

> 面向长期迭代（6个月+）的产品知识体系、功能审计与技能架构

---

# Part A：产品审计

## A.1 用户是谁

| 用户分层 | 画像 | 核心场景 |
|----------|------|----------|
| **主力用户** | 22–35 岁中国 NBA 球迷，每周看球 1–3 次，关注选秀但信息源以中文为主 | "选秀大会快到了，我想提前了解今年有哪些新秀值得关注" |
| **进阶用户** | 深度篮球爱好者，懂数据（PER/TS%/BPM），有自己偏好的球员风格 | "我喜欢攻防一体的侧翼，帮我找类似伦纳德的新秀" |
| **路人用户** | 被社交媒体引流，对 NBA 选秀好奇但认知几乎为零 | "朋友都在聊选秀，给我推荐几个有意思的新人" |

**共同特征**：
- 以中文为第一语言
- 对英文球探报告有阅读障碍
- 需要"降低决策成本"——帮助他们在 60 名新秀中快速锁定 3–5 个值得关注的球员
- 希望通过"我喜欢 XX 类型的球员"这种直觉方式获得推荐，而非填写复杂筛选条件

## A.2 核心需求（按优先级排序）

1. **发现**："帮我找到我可能会喜欢的新秀"
2. **理解**："这个球员到底什么水平？优势和短板是什么？"
3. **比较**："A 和 B 谁更适合我的主队？"
4. **追踪**："我之前关注的球员最近怎么样了？"
5. **炫耀**："我对选秀的了解比朋友多，我想分享我的发现"

## A.3 功能审计：保留 / 删除 / 延后

### ✅ 保留（MVP 核心）

| 功能 | 理由 |
|------|------|
| **3 步入职流程**（位置→球星→天赋/勤奋） | 产品差异化的核心。用"你欣赏谁"替代冷冰冰的筛选器，降低认知门槛。这是整个推荐系统的输入源 |
| **篮球 DNA 生成** | 用户画像的可视化表达。不是功能，是品牌资产——"你的篮球DNA"应该成为用户记住 Next Star 的第一联想 |
| **余弦相似度匹配+推荐** | 推荐引擎本身。算法不需要完美，但匹配逻辑需要让用户感到"有理有据" |
| **AI 球探搜索**（自然语言→球员匹配） | 进阶用户的深度入口。区别于传统 mock draft 网站的核心竞争力 |
| **球员详情页**（概览+数据+能力饼图） | 承载"理解"需求。减少 3 个标签页比增加 10 个功能更有价值 |
| **关注/收藏系统** | 承载"追踪"需求。轻量但不可或缺，是用户回访的理由 |
| **球员能力饼图**（非线性对比放大） | 辅助"理解"。一个让用户秒懂的视觉锚点，比 500 字文字报告更有效 |
| **中文球探报告**（综述+模板摘出） | 消除语言障碍。这是中国用户区别于英文用户的核心价值 |

### ❌ 删除（分散资源、偏离主线）

| 功能 | 理由 |
|------|------|
| **资讯推送页（PushPage）** | 内容聚合是独立产品方向。需要编辑团队、内容源维护、版权合规——这些都是 Next Star 当前不具备也不应该投入的能力。推送资讯会让用户困惑："这到底是选秀工具还是新闻 App？" |
| **文章详情页（ArticlePage）** | 同上。如果未来需要内容，应该引导用户跳转到合作媒体，而不是自建内容系统 |
| **通用视频链接**（YouTube 搜索结果 URL） | 虚假的功能承诺。用户点击"精彩集锦"却跳转到 YouTube 搜索页，体验极差。要么接入真实视频数据，要么删除。不要假装有这个功能 |
| **传奇球星赛季数据代理**（/api/nba/stats） | 入职流程中用户选完球星后，不再需要查看该球星的详细赛季数据。这个端点是为不存在的需求服务的 |

### ⏸️ 延后（有价值但时机未到）

| 功能 | 理由 |
|------|------|
| **职业生涯模拟**（最好/正常/保底） | 概念好但当前实现太粗糙——所有球员用同一套文案模板。需要有真实的历史球员发展轨迹数据支撑才能产生可信的模拟结果。属于 V2 功能 |
| **三年成长计划** | AI 生成的泛化建议（"适应NBA节奏""扩大角色定位""冲击首发"）对任何球员都适用，缺乏个性化。需要结合球队深度图、位置竞争格局等外部数据才有说服力 |
| **最佳/最差模板**（AI 洞察中） | 模板句已从综述中摘出展示在概览页，AI 洞察中的完整模板卡片属于冗余。除非能接入真实的球员相似度数据库（如 BBall-Index 的 comps），否则不值得独立占用一个模块 |
| **联合试训排名展示** | 当前仅展示排名数字，缺乏上下文（总共有多少人参与测试？该球员的排名在什么位置？历史同排名球员的发展如何？）。需要补充参照系数据后才能发挥价值 |
| **主页运营卡片** | 当前是静态内容。真正有价值的是动态运营位——例如"本周选秀行情变化""联合试训后飙升球员"。但动态运营需要持续的编辑投入，当前阶段用静态精选即可 |

## A.4 MVP 必须保留清单

```
入职流程（位置→球星→天赋）
    ↓
篮球 DNA 可视化
    ↓
推荐匹配结果（Top 3）
    ↓
进入主页 ←→ 球员详情 ←→ 球探搜索
    ↑                    ↓
    └── 关注列表 ←──────┘
```

**一句话 MVP 定义**：
> 一个让中国 NBA 球迷通过"我欣赏谁"的直觉方式，发现 2026 年选秀中值得关注的新秀，并用中文理解他们特点的 AI 工具。

## B.2 每个目录的维护说明

### `docs/product/` — 产品决策知识

| 文件 | 为什么存在 | 存什么内容 | 如何维护 |
|------|-----------|-----------|---------|
| `vision.md` | 对齐团队对产品方向的认知，避免迭代迷失 | 一句话产品定义、3/6/12 个月里程碑、北极星指标 | 每月 review 一次，重大 pivot 时立即更新 |
| `user-personas.md` | 确保每个功能决策都有真实的用户场景支撑 | 3 类用户的详细画像、核心场景、痛点、使用频率 | 每季度回访用户后更新，或新发现用户分群时补充 |
| `feature-canvas.md` | 防止功能蔓延（feature creep）——每次想加功能时先查这张表 | 功能清单 + 保留/删除/延后分类 + 每次决策的日期和理由 | 每次功能评审会议后更新 |
| `competitive-audit.md` | 知道对手在做什么，避免重复造轮子或陷入盲区 | 5 个竞品的功能对比表、差异化定位图、机会缺口 | 每季度更新一次，竞品发布重大更新时即时更新 |

### `docs/design/` — 设计规范

| 文件 | 为什么存在 | 存什么内容 | 如何维护 |
|------|-----------|-----------|---------|
| `design-tokens.md` | 组件开发时不用每次都猜测颜色/间距/字体 | Tailwind 扩展 token、暗色主题色板、动效时长与缓动函数 | 设计系统变更时更新，向前兼容 |
| `component-specs.md` | 同一组件多人开发时保持一致性 | 球员卡片、饼图、数据表、标签按钮的尺寸/状态/变体规范 | 新增组件或组件变体时补充 |
| `interaction-patterns.md` | 全局交互一致性——加载状态不应该每个页面不同 | 加载骨架屏、空状态占位图、错误提示、成功 toast 的规范 | 发现新的交互需求时补充 |

### `docs/data/` — 数据知识

| 文件 | 为什么存在 | 存什么内容 | 如何维护 |
|------|-----------|-----------|---------|
| `data-dictionary.md` | 新人接手或 AI 编码时，需要知道每个字段的含义和来源 | 36 名球员所有字段的名称、类型、取值范围、数据来源、更新频率 | 新增字段或数据源变更时同步更新 |
| `tag-taxonomy.md` | 120+ 标签需要分层分类，否则搜索和展示都会混乱 | 标签分为能力层/风格层/风险层/背景层，每层的标签列表 | 新增球员时如引入新标签，在此注册 |
| `translation-guide.md` | 篮球术语翻译需要一致性（"finishing"在不同上下文不能有时译"终结"有时译"得分"） | 术语→固定译法对照表、禁止译法清单、已翻译文本统计 | 发现翻译不一致时修正并记录 |

### `docs/algorithm/` — 算法白盒

| 文件 | 为什么存在 | 存什么内容 | 如何维护 |
|------|-----------|-----------|---------|
| `dna-engine-spec.md` | 权重/基线调优需要可追溯的决策依据 | 算法公式、权重来源说明、基线向量设计原理、调优历史 | 参数变更时记录"为什么改"和"A/B 结果" |
| `matching-spec.md` | 匹配策略（属性80%+embedding20%、位置优先）需要文档化 | 完整匹配流程、权重分配理由、边界条件处理 | 算法策略变更时更新 |
| `scout-pipeline.md` | 3 阶段流水线是系统的核心链路，出问题时需要排查手册 | 每阶段的输入/输出/失败模式/降级策略 | 流水线逻辑变更时更新 |

### `prompts/` — 提示词工程

| 目录 | 为什么存在 | 维护方式 |
|------|-----------|---------|
| `system/` | 当前生产环境使用的提示词。AI 编码时引用这些文件而不是从代码中复制 | 修改提示词时先在此编辑，review 后再同步到代码 |
| `evolution/` | 提示词迭代有历史可追溯。知道 v1 做了什么、v2 改了什么、哪个版本效果好 | 每次提示词变更时，将旧版本归档到此，并更新 changelog |
| `testing/` | 提示词不能被"感觉"评估——需要固定测试用例来量化效果变化 | 每次新增边界 case 时补充测试用例 |

### `skills/` — 技能架构

| 目录 | 为什么存在 | 维护方式 |
|------|-----------|---------|
| `core/` | 产品必须运行的技能。这些技能如果失效，产品就没有价值 | 任何变更需要回归测试 |
| `engagement/` | 提升用户留存和深度的技能 | 按迭代节奏逐步建设 |
| `content/` | 内容相关技能，依赖外部数据源 | 数据源可用时才投入 |
| `growth/` | 增长和传播技能 | 产品稳定后启动 |

## C.2 每个 Skill 的详细规格

---

### Skill 1: DNA Profiling（用户 DNA 画像生成）

| 维度 | 内容 |
|------|------|
| **定位** | Core — 整个推荐系统的输入源，不可移除 |
| **输入** | `selectedPosition` (PG/SG/SF/PF/C)、`selectedStarPlayerIds[]`、`talentType` (talent/hardwork) |
| **处理** | 提取球星属性向量 → 逐元素平均 → 天赋/勤奋权重加权 → 归一化到 [0,100] |
| **输出** | `dnaVector: number[5]`、`dimensions: {label, value}[]`、`description: string`、`positionProfile: string` |
| **依赖数据** | `star-players.ts` (25 名球星 × 5D 属性向量)、`POSITION_BASELINES` (位置基线) |
| **当前状态** | ✅ 已实现 |
| **未来扩展** | ① 支持多球星混合画像（"50% 库里 + 50% 欧文"）；② 历史 DNA 版本对比（"你 3 个月前的 DNA vs 现在"）；③ 好友 DNA 相似度（社交裂变） |

---

### Skill 2: Player Match（球员匹配引擎）

| 维度 | 内容 |
|------|------|
| **定位** | Core — 整个产品的核心价值，不可移除 |
| **输入** | `dnaVector: number[5]`、`selectedPosition: string`、`topN: number`（默认 3） |
| **处理** | 属性向量余弦相似度 (80%) + Embedding 相似度 (20%) → 同位置优先排序 → Top N |
| **输出** | `ProspectMatch[]`：id, name, position, matchScore, attributes |
| **依赖数据** | `2026-draft-database.json`（36 名球员的属性向量 + embedding） |
| **当前状态** | ✅ 已实现 |
| **未来扩展** | ① 风格多样性保证（避免推荐 3 个同类型球员）；② 球队需求匹配（用户指定"我支持火箭队，他们需要中锋"）；③ 历史选秀数据库扩展（不止 2026，扩展到 2025/2024 届） |

---

### Skill 3: Player Search（球探搜索）

| 维度 | 内容 |
|------|------|
| **定位** | Core — 进阶用户的深度入口，与 Player Match 互补 |
| **输入** | `query: string`（中文自然语言，如"3D 侧翼，三分准，能换防"）、可选的 `dnaVector` 混合 |
| **处理** | Phase 1: LLM 将查询转为 5D 查询向量 → Phase 2: 余弦相似度排序 (80% 属性 + 20% embedding) → Phase 3: LLM 生成推荐解释 |
| **输出** | `recommendations[]`（matchScore + reason + strengths + risks）+ `summary` |
| **依赖数据** | `2026-draft-database.json`、DeepSeek API、`scout-vectorizer.md` 提示词、`scout-explainer.md` 提示词 |
| **当前状态** | ✅ 已实现 |
